# VerdeAI v9.0 — Secure one-image AI pilot recommendation

## Decision

**Prepared, not approved. Do not enable paid rendering yet.**

The static frontend, disabled backend scaffold, confirmation fields, one-image limit, image-size checks, timeout path, provider abstraction, mock fallback, kill switch and test mode are suitable foundations. The missing approval evidence is a genuine v9.0 physical-phone calibration pass.

## Why calibration comes first

The calibrated usable area, keep-clear zones, access line, opportunity point and marker 5 are the only honest spatial instructions available before computer vision is connected. A real render should not receive a property photo until the owner has confirmed those controls work reliably with one finger.

## Smallest later pilot

- Invited users only.
- One property photo.
- One selected future.
- One confirmed render per session.
- No render-all-six action.
- Server-side provider key only.
- Explicit photo/privacy confirmation.
- Explicit maximum-cost confirmation.
- Input resizing and metadata removal.
- Provider calls disabled by default.
- Hard kill switch and test mode on by default.
- Calibrated overlay returned as the free fallback on timeout or failure.
- No sensitive photo content in application logs.

## Decisions still requiring owner approval

1. Rendering provider.
2. Backend host.
3. Total pilot budget cap.
4. Maximum per-render authorised cost.
5. Provider retention/deletion policy acceptance.
6. Whether invited testers may upload photos to the provider.

## Recommended next gate

Do not choose a provider merely because the scaffold lists one. First complete the physical Android evidence checklist. Then compare current official provider pricing, editing quality, retention terms and latency immediately before the owner approves the pilot.
